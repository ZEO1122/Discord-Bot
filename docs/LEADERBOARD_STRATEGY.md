# Leaderboard Strategy

이 문서는 현재 프로젝트에서 리더보드/상금 운영을 어떤 방향으로 설계할지 정리한 의사결정 문서다.

## 1. 현재 상태

현재 저장소는 **브리핑 자동 발행 시스템**으로서의 MVP를 거의 갖춘 상태다.

완료된 범위:
- concept 브리핑 자동 게시
- trend 브리핑 자동 게시
- 운영 문서 정리

현재 기본 런타임:
- **Google Apps Script + Discord Webhook**

## 2. 리더보드 논의가 필요한 이유

다음 단계는 **참여 유도형 학습 운영**이다.

핵심 목표:
- 주니어 부원의 참여율을 높인다.
- 브리핑을 읽고 실제로 학습했는지 확인한다.
- 퀴즈 결과를 기반으로 리더보드를 만들고, 이후 상금 등 인센티브 구조로 연결한다.

## 3. 퀴즈 형식 비교

### 단답형
- 자동 채점이 쉽다
- accepted keyword 기반 판정이 가능하다
- 리더보드와 연결하기 좋다

### 서술형
- 실제 이해도를 더 잘 본다
- 자동 채점이 어렵고 공정성 설명이 어렵다

## 4. 리더보드 기준 비교

### 정답 기준
- `unique_correct_quiz_count`
- `first_try_correct_count`
- `total_correct_attempts`

장점:
- 학습 성과 중심
- 상금 기준으로 설명하기 쉽다

### 제출 횟수 기준
- `total_attempts`

장점:
- 참여 유도에 강함

단점:
- 양 위주로 흐를 수 있음

## 5. 현재 구조의 한계

현재 기본 운영은:
- GAS + Discord Webhook

이 구조로 가능한 것:
- 브리핑 자동 게시
- 채널별 weekly trend 게시

이 구조로 어려운 것:
- Discord 안에서 실시간 답변 제출
- 사용자별 실시간 기록 저장
- 리더보드 즉시 갱신

## 6. 추천 방향

현재 조건에서는 아래가 가장 현실적이다.

1. 메인 자동화는 GAS 유지
2. 퀴즈는 게시 중심으로 운영
3. 실제 리더보드/상금은 나중에 별도 상시 런타임을 도입할 때 재설계

## 7. 향후 확장 후보

- Discord bot 상시 운영
- Cloudflare Workers 기반 interaction 처리
- 외부 폼/시트 기반 제출 수집
- 별도 DB 기반 통계/리더보드

## 8. 현재 권장 결론

- 지금 당장은 브리핑 자동화에 집중
- 리더보드 기능은 후속 확장으로 분리
