# PRD — AI 학술동아리 Discord 학습봇

## 1. 배경

학술동아리 서버에는 정보가 흩어지기 쉽고, 꾸준한 학습 흐름이 만들어지기 어렵다.  
특히 딥러닝 기초 개념과 최신 연구 동향은 읽어도 남는 것이 적고, 개인별 학습 참여를 추적하기도 어렵다.

이 프로젝트의 목표는 Discord 안에서 다음을 자연스럽게 이어주는 것이다.

- 짧고 신뢰 가능한 브리핑
- 바로 풀 수 있는 퀴즈
- 개인별 응답 기록과 피드백
- 반복 학습과 참여 동기

## 2. 목표

### 제품 목표
- 동아리 구성원이 매일 1개 이상의 학습 콘텐츠를 소비하도록 돕는다.
- 최신 연구 동향을 “너무 어렵지 않게” 요약해서 전달한다.
- 퀴즈를 통해 개념 이해를 빠르게 점검한다.
- 개인별 참여도와 정답률을 기록한다.

### 운영 목표
- 게시물 생성과 스케줄링을 자동화한다.
- 관리자 검수를 거쳐 품질을 유지한다.
- 출처와 브리핑 기록을 남겨 재사용 가능하게 한다.

## 3. 핵심 사용자

### 3.1 일반 동아리원
- 브리핑을 읽고 빠르게 핵심만 파악하고 싶다.
- 퀴즈를 풀고 바로 정답/오답 피드백을 받고 싶다.
- 내가 얼마나 참여했는지 보고 싶다.

### 3.2 운영진/스터디 리더
- 하루/주 단위로 콘텐츠를 배포하고 싶다.
- 최근 동향이 사실에 기반했는지 검수하고 싶다.
- 참여율과 정답률을 보고 다음 주제를 기획하고 싶다.

## 4. 핵심 사용자 시나리오

### 시나리오 A — 오늘의 브리핑
1. 사용자가 `#daily-briefs` 채널에서 오늘의 브리핑을 본다.
2. 핵심 설명, 왜 중요한가, 쉬운 용어, 출처를 확인한다.
3. 버튼을 눌러 퀴즈를 푼다.
4. 개인 피드백을 즉시 받는다.

### 시나리오 B — 내 기록 확인
1. 사용자가 `/stats me` 명령을 실행한다.
2. 총 응답 수, 정답 수, 최근 참여 streak, 분야별 강/약점을 본다.

### 시나리오 C — 관리자 게시
1. 관리자가 생성된 초안을 검수한다.
2. 승인 후 특정 채널에 게시한다.
3. 게시 결과와 메시지 ID, content ID를 저장한다.

## 5. 범위

## In scope
- Discord 브리핑 게시
- 퀴즈 제출
- 사용자별 응답 기록
- 출처 표시
- 관리자 게시/재게시
- 리더보드는 현재 범위에서 제외
- Linux 서버 운영 기준 문서화

## Out of scope (초기)
- 완전 자동 논문 이해 평가
- 음성/화상 기반 인터랙션
- 고급 개인화 추천
- 외부 LMS 연동
- 대규모 멀티 서버 SaaS 구조

## 6. MVP 정의

MVP는 아래를 만족하면 된다.

- 분야 3개 이상 지원: `DL Basics`, `NLP`, `CV`
- 하루 1회 자동 게시 가능
- 게시물마다 퀴즈 1개 이상 연결
- 사용자별 시도 횟수와 정답 여부 기록
- `/stats me`, `/leaderboard`, `/brief today` 명령 제공
- 출처를 게시물에 표시
- 운영진이 수동 승인 후 게시 가능

## 7. 기능 요구사항

### 7.1 콘텐츠 브리핑
- 제목
- 한 줄 요약
- 핵심 설명
- 왜 중요한가
- 쉬운 용어 3개
- 토론용 질문
- 출처 1~3개

### 7.2 퀴즈
- 객관식 우선
- 단답형은 선택적
- 힌트
- 해설
- 응답 제한/마감 시간(선택)

### 7.3 사용자 기록
- 누가
- 어떤 퀴즈에
- 몇 번째로
- 언제
- 무엇을 제출했고
- 맞았는지/틀렸는지
- 점수는 얼마인지

### 7.4 관리자 기능
- 게시 승인
- 재게시
- 초안 보관
- 중복 주제 방지
- 출처 없는 초안 차단 또는 수동 승인

## 8. 성공 지표

### 1차 지표
- 주간 활성 응답자 수
- 브리핑당 응답자 수
- 첫 시도 정답률
- 주간 재방문율

### 2차 지표
- 분야별 참여도
- 평균 풀이 시간
- 출처 클릭률
- 오답 재도전 비율

## 9. 비기능 요구사항

- 브리핑 본문은 초보자도 읽을 수 있어야 한다.
- Discord 채널이 과도하게 시끄러워지지 않아야 한다.
- 정답/피드백은 개인에게 우선 제공한다.
- 관리자 개입 없이도 최소 운영이 가능해야 한다.
- 장애 시 하루 게시 실패를 추적할 수 있어야 한다.

## 10. 출시 기준

다음 조건을 만족하면 MVP 출시 가능 상태로 본다.

- 일일 게시 자동화 성공
- 퀴즈 제출/기록이 안정 동작
- 사용자 통계 확인 가능
- 관리자 재게시 가능
- 로그/재시도/기본 예외 처리가 되어 있음
